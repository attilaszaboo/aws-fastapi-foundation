# ECS Fargate + ALB + CDK Deployment Plan

This document is a practical execution plan for deploying this repository's FastAPI container to AWS using:

- Amazon ECR for container images
- Amazon ECS on AWS Fargate for container runtime
- Application Load Balancer (ALB) for public HTTP/HTTPS access
- AWS CDK for infrastructure as code
- GitHub Actions for CI and release publishing

The goal is a public endpoint that a client can call with `curl`, while keeping the implementation small and production-shaped.

## Current state of this repository

Already implemented:

- FastAPI app with `/hello`, `/health`, and `/version`
- Dockerfile that runs the app on port `8000`
- GitHub Actions CI workflow that lints, tests, and builds the image
- GitHub Actions release workflow that manually pushes the image to Amazon ECR
- GitHub Actions deploy workflow that manually promotes a chosen image tag to ECS through CDK
- AWS infrastructure definition with CDK
- ECS service on Fargate behind an ALB
- CloudWatch log group configuration in infrastructure
- Deployment update flow from ECR image tag to running ECS service

Not yet implemented:

- HTTPS with ACM and DNS
- Private-subnet task placement
- richer application behavior

## Target architecture

This is the target AWS architecture for the first real deployment:

1. A VPC across at least two Availability Zones.
2. Public subnets for the ALB.
3. Private subnets for ECS Fargate tasks.
4. An ECS cluster.
5. A Fargate task definition that runs the Docker image from ECR.
6. An ECS service that keeps at least one task running.
7. An Application Load Balancer with a listener forwarding traffic to the ECS service.
8. A CloudWatch log group for container logs.
9. Security groups:
   - ALB allows inbound `80` from the internet.
   - ECS tasks allow inbound `8000` only from the ALB.
10. Optional later:
   - HTTPS on `443` with ACM certificate and Route 53 domain.

## Recommended delivery order

Do this in two stages:

1. Deploy HTTP only first.
   - Use ALB on port `80`.
   - Confirm the service works with `curl`.
   - Keep the scope small.

2. Add HTTPS and a custom domain later.
   - Use ACM certificate.
   - Use Route 53 alias record to the ALB.

This keeps the first deployment simpler and reduces the number of AWS services you need to learn at once.

## Phase 0: Preconditions

### Objective

Confirm your local machine and AWS account are ready for CDK-based deployment.

### Steps

1. Verify AWS CLI credentials work.

```bash
aws sts get-caller-identity
```

Expected result:

- You get your AWS account, ARN, and user or role identity.

2. Decide which AWS region you will use.

Recommendation:

- Use a single region consistently for ECR, ECS, ALB, CloudWatch, and CDK.
- Example: `us-east-1` or the region closest to you.

3. Install Node.js if it is not already installed.

Reason:

- The AWS CDK CLI is distributed via npm.

4. Install the AWS CDK CLI and verify it works.

Official AWS CDK getting started docs recommend:

```bash
npm install -g aws-cdk
cdk --version
```

5. Confirm Docker works locally.

```bash
docker --version
docker build -t aws-fastapi-foundation:local .
```

6. Make sure the ECR repository already used by your release workflow exists in the same AWS account and region you plan to deploy into.

### Success criteria

- `aws sts get-caller-identity` works.
- `cdk --version` works.
- `docker build` works.
- You know the target AWS region.
- Your ECR repository exists and already contains at least one pushed image tag.

## Phase 1: Add CDK project scaffolding

### Objective

Create a dedicated CDK app inside this repository without disturbing the existing FastAPI app.

### Recommendation

Keep infrastructure code in a separate directory, for example:

```text
infrastructure/
```

Suggested structure:

```text
infrastructure/
  app.py
  cdk.json
  requirements.txt
  source.bat
  ecs_fargate_stack.py
```

### Steps

1. Create the CDK app in Python.

Example flow:

```bash
mkdir infrastructure
cd infrastructure
cdk init app --language python
```

2. Create and activate the CDK virtual environment generated by `cdk init`.

3. Install the CDK libraries you need.

You will likely need:

- `aws-cdk-lib`
- `constructs`

The `aws-cdk-lib` package contains the ECS, EC2, ELBv2, IAM, and CloudWatch constructs.

4. Commit the CDK scaffold before heavy customization.

Reason:

- You want a clean checkpoint before adding actual infrastructure resources.

### What you should learn in this phase

- CDK app vs stack
- `cdk synth`
- `cdk diff`
- `cdk deploy`

### Success criteria

- `cdk synth` works from the `infrastructure/` directory.
- The repo contains a separate CDK app dedicated to infrastructure.

## Phase 2: Bootstrap CDK in your AWS account

### Objective

Prepare your AWS environment so CDK can deploy stacks.

### Steps

1. Bootstrap the target account and region.

Example:

```bash
cd infrastructure
cdk bootstrap aws://ACCOUNT_ID/REGION
```

You can find `ACCOUNT_ID` with:

```bash
aws sts get-caller-identity
```

2. If you use named AWS CLI profiles, specify the profile explicitly.

Example:

```bash
cdk bootstrap aws://ACCOUNT_ID/REGION --profile your-profile
```

### Notes

- CDK bootstrap creates supporting resources in your AWS account for CDK deployments.
- This is a one-time setup per account and region, unless you intentionally change the bootstrap configuration.

### Success criteria

- `cdk bootstrap` finishes successfully.
- You understand that CDK deployments are backed by CloudFormation.

## Phase 3: Define the minimal infrastructure stack

### Objective

Create a first deployable stack that runs one copy of your FastAPI container behind an ALB.

### Recommended implementation strategy

Use the higher-level CDK construct:

- `ApplicationLoadBalancedFargateService`

Reason:

- It is the simplest clean path for a learning project.
- It creates and wires the ALB, target group, listener, Fargate service, and other common pieces.
- You can always refactor to lower-level constructs later if you want more control.

### Core resources to define

1. VPC
2. ECS cluster
3. CloudWatch log group
4. Fargate service using image from ECR
5. Public ALB
6. Security defaults that keep the task private behind the ALB

### Important design decisions

1. Use `desired_count=1` at first.
   - Keep cost and complexity down.

2. Start with small task sizing.
   - Example starting point: `cpu=256`, `memory_limit_mib=512`
   - Increase later only if needed.

3. Put Fargate tasks in private subnets.
   - The ALB is public.
   - The container tasks do not need to be directly internet-exposed.

4. Health check path should be `/health`.
   - Your app already provides this endpoint.

5. Container port should be `8000`.
   - This matches the Dockerfile.

### Example implementation checklist

Your stack should define:

1. A VPC with at least two Availability Zones.
2. An ECS cluster in that VPC.
3. A reference to your existing ECR repository.
4. A Fargate service using an image tag that you choose.
5. ALB health checks pointing at `/health`.
6. CloudWatch logging for the container.
7. Stack outputs for:
   - ALB DNS name
   - ECS cluster name
   - ECS service name

### What not to optimize yet

- Multi-environment deployment
- Blue/green deployment
- WAF
- HTTPS
- Autoscaling
- Custom domain
- Secrets Manager integration

Keep the first stack minimal and working.

### Success criteria

- `cdk synth` succeeds.
- `cdk diff` shows the expected infrastructure.
- You can explain each major resource in the stack and why it exists.

## Phase 4: Parameterize the image source

### Objective

Make the deployed service point to a specific ECR image tag.

### Recommendation

Start simple:

- Hardcode a known good image tag in the stack during initial learning.

Then improve:

- Move the tag to a CDK context value, environment variable, or stack parameter.

Example idea:

- `image_tag=0.1.1`

Why this matters:

- Your GitHub release workflow already pushes versioned images to ECR.
- ECS needs a concrete image reference, not just "some image exists."

### Practical advice

During first deployment, avoid making CDK dynamically discover the latest image.

Reason:

- "Latest" is ambiguous and makes deployments less reproducible.
- Explicit tags are easier to reason about and easier to discuss in interviews.

### Success criteria

- The stack deploys a known image tag from your ECR repository.
- You know exactly which app version is running in ECS.

## Phase 5: Deploy the first stack

### Objective

Create the real AWS infrastructure and start the service.

### Steps

1. Run synthesis first.

```bash
cd infrastructure
cdk synth
```

2. Review differences.

```bash
cdk diff
```

3. Deploy.

```bash
cdk deploy
```

4. Watch CloudFormation events if deployment fails.

You can use:

- CDK terminal output
- CloudFormation console
- ECS console

### Expected result

At the end of deployment, CDK should output the ALB DNS name.

Example shape:

```text
http://my-load-balancer-123456789.region.elb.amazonaws.com
```

### Success criteria

- CloudFormation stack reaches `CREATE_COMPLETE`.
- ECS service reaches stable state.
- At least one task is healthy in the target group.
- You receive an ALB DNS name as stack output.

## Phase 6: Verify the running service end to end

### Objective

Prove the service is reachable from the public internet and that the ALB is forwarding correctly.

### Steps

1. Test the health endpoint.

```bash
curl http://ALB_DNS_NAME/health
```

Expected:

```json
{"status":"ok"}
```

2. Test the hello endpoint.

```bash
curl http://ALB_DNS_NAME/hello
```

3. Test the version endpoint.

```bash
curl http://ALB_DNS_NAME/version
```

4. Open the ALB DNS name in a browser and inspect:

- `/docs`
- `/health`

5. Inspect CloudWatch logs for the running task.

Verify:

- The container starts cleanly.
- Requests appear in logs if you have request logging enabled later.

### If it does not work, check these first

1. ALB security group:
   - Inbound `80` from `0.0.0.0/0`

2. ECS service security group:
   - Inbound `8000` only from the ALB security group

3. Target group health check:
   - Path `/health`
   - Correct port

4. Container port mapping:
   - `8000`

5. ECR image tag:
   - The tag in CDK matches an image that exists

### Success criteria

- `curl` to the ALB works from your machine.
- ECS task is healthy.
- Target group shows healthy targets.
- Logs are visible in CloudWatch.

## Phase 7: Improve the stack for production-shaped learning

### Objective

Add the first round of improvements that better reflect real-world deployment, without over-engineering.

### Improvements to add

1. Enable HTTPS with ACM and Route 53.

Add:

- Public hosted zone in Route 53
- ACM certificate in the same region as the ALB
- ALB listener on `443`
- Optional redirect from `80` to `443`

Why:

- Public APIs should use HTTPS.
- This materially improves the realism of the deployment.

2. Add ECS service autoscaling.

Start simple:

- Minimum tasks: `1`
- Maximum tasks: `2` or `3`
- Scale on CPU utilization

Why:

- It teaches a real production concept without much complexity.

3. Add access logging later if you want deeper observability.

Possible later additions:

- ALB access logs
- Structured app logging
- CloudWatch dashboard

4. Tighten IAM and tagging.

Add tags like:

- `Project=aws-fastapi-foundation`
- `Environment=dev`
- `Owner=your-name`

### Success criteria

- HTTPS works with a certificate.
- The stack remains understandable.
- You can explain how traffic reaches the container end to end.

## Phase 8: Integrate deployment with GitHub Actions

### Objective

Extend the release workflow so a manual release can update the running ECS service, not just push to ECR.

### Recommended rollout

Do this after manual deployment works.

Reason:

- First learn the system manually.
- Then automate it.

### Options

#### Option A: GitHub Actions runs `cdk deploy`

Pros:

- Simple mental model
- Infrastructure and app deployment stay in one place

Cons:

- GitHub Actions needs permission to deploy CloudFormation/CDK-managed infrastructure
- Broader IAM permissions than only pushing to ECR

#### Option B: GitHub Actions updates only the ECS service to use a new image tag

Pros:

- Narrower deployment permissions
- Good if infrastructure changes are infrequent

Cons:

- You must separately manage infrastructure changes
- App deployment and infrastructure deployment become two workflows

### Recommendation

For this project:

- Start with Option A during learning, if you are comfortable with the IAM setup.
- If that feels too broad, keep infrastructure manual at first and automate later.

### Concrete release target

Your release workflow should eventually do this:

1. Build and push image to ECR.
2. Deploy or update the CDK stack with the chosen image tag.
3. Run a post-deploy smoke test against `/health`.

### Success criteria

- A manual release produces a new running version in ECS.
- You can verify the deployed version through `/version`.

## Phase 10: Cleanup discipline

### Objective

Avoid leaving billable resources running unintentionally.

### Steps

When you are done experimenting:

1. Destroy the CDK stack if you no longer need it.

```bash
cd infrastructure
cdk destroy
```

2. Check for leftovers:

- ECR images
- CloudWatch log groups
- Route 53 hosted zone
- ACM certificates

### Success criteria

- You know how to cleanly remove the stack.
- You understand which resources may continue to incur cost if left behind.

## Suggested implementation backlog

Execute in this order:

1. Verify AWS identity, region, Docker, and CDK CLI.
2. Create the Python CDK app under `infrastructure/`.
3. Bootstrap CDK in your target account and region.
4. Define a minimal ECS Fargate + ALB stack using `ApplicationLoadBalancedFargateService`.
5. Reference a fixed ECR image tag.
6. Deploy with `cdk deploy`.
7. Validate with `curl` against the ALB DNS name.
8. Add HTTPS and domain later.
9. Add autoscaling later.
10. Extend GitHub Actions to deploy the service after pushing the image.

## Common mistakes to avoid

1. Using `latest` everywhere.
   - Prefer explicit image tags.

2. Trying to automate CDK deploy before manual deploy works.
   - First make it work manually.

3. Exposing the ECS task directly to the internet.
   - Put the ALB in front.

4. Forgetting the health check path.
   - Your service should use `/health`.

5. Mixing regions.
   - Keep ECR, ECS, ALB, and CDK in the same region unless you have a strong reason not to.

6. Starting with HTTPS, custom domain, autoscaling, and CI/CD automation all at once.
   - Sequence the work.

## What "done" looks like for the first milestone

The first milestone is complete when all of the following are true:

1. CDK defines the AWS infrastructure in code.
2. The stack deploys successfully with `cdk deploy`.
3. The ECS service pulls the image from ECR.
4. The ALB exposes a public DNS name.
5. `curl http://ALB_DNS_NAME/health` returns a healthy response.
6. `curl http://ALB_DNS_NAME/version` shows the deployed version.
7. You can see the running container logs in CloudWatch.
